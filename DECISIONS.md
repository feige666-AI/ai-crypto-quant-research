# 技术决策记录

## 2026-07-16：核心逻辑以 Python 标准库为主

决策：数据读取、校验、策略、回测、CLI 和报告使用标准库；只为图表引入 matplotlib。

原因：降低认知和安装复杂度，让作品重点落在可解释的计算流程。matplotlib 会传递安装其底层依赖，但项目没有直接使用 pandas 或数据框架。

## 2026-07-16：使用 argparse

决策：CLI 使用标准库 `argparse`。

原因：四个子命令规模有限，`argparse` 足够稳定，也符合“不过度堆框架”的定位。

## 2026-07-16：使用固定种子合成数据

决策：默认样例由 `scripts/generate_sample_data.py` 以 300 行、种子 42 生成。

原因：结果可重复、可公开、没有账户或数据授权风险。影响是结果不能用于市场结论。

## 2026-07-16：保持单资产只做多

决策：只支持一个资产、全仓买入和全仓卖出。

原因：避免做空、保证金、强平和组合分配分散教学重点。该决定可在未来独立阶段重新评估，但不属于当前作品集范围。

## 2026-07-16：不连接交易所

决策：不加入任何交易所 API、密钥、登录或订单路径。

原因：保持离线可测试，避免真实资金和隐私风险。

## 2026-07-16：当前收盘价形成信号并模拟成交

决策：信号使用当前及之前数据，买入价为 `close × (1 + slippage)`，卖出价为 `close × (1 - slippage)`。

原因：规则简单且容易复现。代价是存在同 bar 时点偏差，文档必须明确，不能直接用于真实交易。

## 2026-07-16：双边手续费按成交金额计算

决策：入场和退出都按成交金额收取手续费；全仓买入数量预留入场手续费。

原因：保证现金不会因费用变成负数，并让成本计算一致。

## 2026-07-16：默认期末强制平仓

决策：`close_position_at_end=true` 为默认，并提供 `--no-close-position-at-end`。

原因：默认样例能形成完整交易和稳定指标；关闭后按末价估值并明确标记未平仓，不把它算作完成交易。

## 2026-07-16：使用 matplotlib 输出静态 PNG

决策：生成价格/信号、资金曲线和回撤三张 PNG。

原因：PNG 可直接在 GitHub README 展示，且无需 Web 前端或运行服务。

## 2026-07-16：不加入数据库和 Web 前端

决策：输入和输出保持 CSV、JSON、PNG 文件。

原因：项目数据量小、完全本地，数据库和 Web 服务不会提升核心学习价值，反而增加无关复杂度。

## 2026-07-16：GitHub API 发布作为网络兼容路径

决策：当前执行环境的 `git clone` 被 TLS/HTTP 403 网络策略拦截后，使用已登录 GitHub CLI 的官方 API 读取最新 `main`、创建工作分支和发布 Git 对象。

原因：仓库和账号权限已由 GitHub 插件验证，API 路径仍会生成以 `main` 为父提交的真实分支、commit 和 PR，不改写历史。
